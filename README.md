# 自动构建服务

基于 GitHub Actions Workflow 功能，提交代码时为提交添加指定格式的 TAG，即可自动触发构建流程。全程利用 GitHub 服务器资源生成目标文件，无需在本地配置复杂构建环境，高效完成产物输出。

## 核心优势
- 提交触发：代码推送或 PR 提交后自动启动构建，无需手动触发，减少操作成本。
- 环境统一：依托 GitHub 标准化服务器环境，规避本地环境版本差异、依赖缺失等导致的构建失败问题。
- 产物集中：所有构建输出文件统一存放于 `out` 目录，结构清晰，便于后续打包、下载或部署操作。

## 目录结构说明
```bash
.
├── demo                   # 快速上手示例目录
│   ├── hello.c            # 示例源代码文件（基础可执行程序示例）
│   ├── test1.sh           # 示例构建脚本1：执行后编译 hello.c 生成 hello 二进制文件，输出至 out 目录
│   └── test2.sh           # 示例构建脚本2：备用构建逻辑演示
├── LICENSE                # 开源许可文件（项目授权说明）
├── out                    # 构建产物输出目录：所有构建生成的目标文件统一存放于此
├── README.md              # 项目说明文档（含使用指南、目录说明等）
├── run -> ./demo/test1.sh # Workflow 默认执行脚本（软链接）：默认指向示例脚本，建议替换为实际项目构建脚本
└── tools
    ├── ln2run.sh          # 快速链接工具：用于便捷修改 Workflow 默认执行脚本
    └── release-notes-generator.sh  # 自动生成 Release 说明脚本：构建 Release 时自动调用，请勿修改
```

## 快速使用
1. 配置执行脚本：通过 `./ln2run.sh your_script.sh` 命令，将 `run` 软链接指向你的实际构建脚本（确保脚本可执行且输出产物至 out 目录）。
2. 提交代码修改：将上述配置修改添加到本地提交（执行 `git add .` + `git commit -m "配置构建脚本链接"`）。
3. 添加合规 TAG：为当前提交添加 TAG，格式必须以 `v` 开头，后跟版本号（支持主版本号、次版本号、修订号，示例：`v1.0`、`v1.2.3`、`v2.0.0-beta`）。
4. 触发自动构建：将本地提交（含 TAG）推送至 GitHub 仓库，系统自动触发 Workflow 构建流程，完成后产物存放于 `out` 目录。
